Command line use

Self-signing enables its use via command line, providing signature, batch signing, and hashes creation and verification capabilities.

The executable or script to use for the execution of Autofirma by command line varies depending on the system:

  • Windows
    • The installation process copies in the Autofirma directory the executable "Autofirma CommandLine" and, if possible, includes the directory in the path. Use this executable from command line to perform the various operations that are supported.
  • Linux
    • The installation process generates in the directory "/usr/bin" the script "Autofirma". Use this script to run the application by providing you directly with the parameters of the operation you want.
  • macOS
    • For the use of the command-line application a script "AutoCommandLine" is created within the application in the application directory. You can run this script by accessing the internal "Resources" directory of the application (commonly, "/Applications/Autofirma.app/Contents/MacOS"). It is important that the parameters that may contain spaces are passed in the command between simple quotation marks.

For example, you can run Autofirma and list the different operations available through the command line with the "-help" command as follows:

In Windows:
  • Autofirma CommandLine – help
In Linux:
  • /usr/bin/Autofirma – help
On Mac:
  • /Applications/Autofirma.app/Contents/MacOS/AutoCommandLine – help

This would result in:

Document signing tool.

Syntax: Self-signing cmd [options...]

Cmd commands:

  sign (file signature)
  cosign (file cosignature)
  countersign (file counter)
  listaliases (list available aliases in stock)
  verify (opens a panel with validation information for a signature or certificate)
  batchsign (signature/multi-signature lots of files)
  createdigest (creation of digital fingerprints with graphical interface)
  checkdigest (fingerprint check with graphical interface)

Use "Autofirma cmd -help" to view the "cmd" command syntax

To list the configuration options of any of the operations, you can use the "-help" command over any of them. For example:

  • Autofirma CommandLine sign –help

This would result in:

The config configuration parameter allows you to be provided with a list of configuration properties concatenated by "\n". Each of these properties will have the key format=value.

An example of a configuration parameter would be:

  • -config "format=XAdES Enveloping\nsignerEmbassyRoles=Director\nxadesNamespace=http://uri.etsi.org/01903/v1.4.1#

Configuration options vary from one signature format to another. The following describe the configuration options for each signature format.

CAdES Signature Configuration Options

The configuration properties supported by CAdES signatures are listed below. Some of these options may not have effect in cases of signature where they do not apply, as in the case of counter-affirmations.

  • mode:
    • explicit: The resulting signature shall not include the signed data. If the mode property is not indicated, this behavior is automatically configured.
    • implicit: The resulting signature shall not include the signed data. The resulting signature shall include internally a copy of the signed data. The use of this value could generate large signatures. In the co-signatures, this property will be ignored if the data were already contained in the original signature or if the data is not provided.
  • contentTypeOid:
    • Identifier of the type of data signed.
  • contentDescription:
    • Textual description of the type of data signed.
  • includeMimeTypeAttribute:
    • true: Includes in the signature the attribute id-aa-ets-mimeType defined in CAdES v2.1.1 and above (valid in baseline signatures). The mimetype value will be taken from the mimetype property or, if not included, it will be extrapolated from the contentTypeOid property. If this is not possible, these values will also be obtained from one of the previous signatures (in the case of co-signatures). If this is not possible, the type of data analysis (which might not be correct or concrete) will be obtained. If the type cannot be obtained in any way, the mimetype application/octet stream shall be used.
    • false: Does not include the id-aa-ets-mimeType attribute in the signature. Default value.
  • mimeType:
    • MIME-Type of data to be signed. It is only included when the property is indicated.
  • policyIdentifier:
    • Signature policy identifier, required to generate CAdES-EPES signatures.
  • policyIdentifierHash:
    • Base Chain 64 with the fingerprint of the signature policy. It is mandatory to indicate this paramount property if Identifier has also indicated, as it is also mandatory to give value to the policyIdentifierHashAlgorithm property.
  • policyIdentifierHashAlgorithm:
    • SHA1: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA1 algorithm.
    • SHA-256: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-256 algorithm.
    • SHA-384: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-384 algorithm.
    • SHA-512: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-512 algorithm.
  • policyQualifier:
    • URL accessible to the document (usually PDF) containing a textual description of the signature policy. This property is optional even if you want to generate CAdES-EPES signatures.
  • includeOnlySignningCertificate:
    • true: Indicates that only the certificate of the signatory should be included in the signature.
    • false: Indicates that the entire certification chain of the signatory certificate should be included in the signature. Default value.
  • policyQualifier:
    • URL accessible to the document (usually PDF) containing a textual description of the signature policy. This property is optional even if you want to generate CAdES-EPES signatures.
  • signatureProductionCity:
    • It adds to the firm a field with the city in which the signature is made. The encoding should be UTF-8.
  • signatureProductionPostalCode:
    • Add to the firm a field with the postal code where the signature is made. The encoding should be UTF-8.
  • signatureProductionCountry:
    • Add to the firm a field with the country in which the signature is made. The encoding should be UTF-8.
  • signerClaimedRoles:
    • He adds fields to the firm with the charges attributed to the signatory. The charges must be separated with the character "|" (and this cannot be in the text itself of any charge).
  • commitmentTypeIndications:
    • Number of commitmentTypeIndications to be declared. These are the reasons for the signing. Specific values are specified with TypeIndicationnIdentifier and TypeIndicationnDescription, where ‘n’ goes from 0 to the value indicated on this property minus 1.
  • commitmentTypeIndicationnIdentifier:
    • 1: It states that the number n (counting from scratch) TypeIndications is "Proof of Origin".
    • 2: It states that the number n (counting from scratch) TypeIndications is "Reception Test".
    • 3: It states that the number n (counting from scratch) TypeIndications is "Delivery Test".
    • 4: Sets that the number n (counting from scratch) is "Shipping Test".
    • 5: It states that the number n (counting from scratch) TypeIndications is "Approval Test".
    • 6: Establishes that the number n (counting from scratch) TypeIndications is "Proof of Creation".
  • commitmentTypeIndicationnCommitmentTypeQualifiers:
    • List of textual indicators separated by '|' character that are provided as additional qualifiers of the CommitmentTypeIndication number n (optional attribute). They're usually OID. The items in the list cannot contain the '|' character (as it is used as a separator).
  • commitmentTypeIndicationnIdentifier:
    • true: The SigningCertificateV2 attribute shall be included in the signature.
    • false (or other value): The SigningCertificate attribute shall be included in the signature.
    • Unspecified: The SigningCertificate will be included if the signature uses a signature algorithm SHA1 and SigningCertificateV2 for the other algorithms.
  • allowSignLTSignature:
    • true: Longevo file signatures are allowed to be co-signed/countersigned, even though they will not be valid thereafter.
    • false (or other value): Co-signing/counter-signing of longevo file signatures is not permitted.

XAdES sign-up options

The configuration properties supported by the XAdES signatures are listed below. Some of these options may not have effect in cases of signature where they do not apply, as in the case of counter-affirmations.

  • insertEnvelopedSignatureOnNodeByXPath:
    • Expression XPath (v1) indicating the node under which the signature node must be inserted in the case of a Enveloped signature. If the expression returns more than one node, only the first one is used. If the expression does not return nodes or is poorly constructed, an exception will be launched. This property only has effect on Enveloped signatures.
  • useManifest:
    • true: Use a XMLDSig Manifest with signature references instead of signing these references directly. It is ignored in the co-signature operation. This allows checking the destination and fingerprints of the references to be optional.
    • false: Generate the signatures normally, without Manifest (default behavior).
  • urin:
    • URI which refers to the data to be signed within a manifest signature. ‘n’ indicates the reference number of those who want to sign, starting with ‘1’.
  • precalculatedHashAlgorithm:
    • SHA1: It indicates that the fingerprints of the data referenced in the manifest were calculated using the SHA1 algorithm.
    • SHA-256: It indicates that the fingerprints of the data referenced in the manifest were calculated using the SHA-256 algorithm.
    • SHA-384: It indicates that the fingerprints of the data referenced in the manifest were calculated using the algorithm SHA-284.
    • SHA-512: It indicates that the fingerprints of the data referenced in the manifest were calculated using the SHA-512 algorithm.
  • mdn:
    • Base 64 with the fingerprint of the data associated with the reference ‘n’. The fingerprint algorithm shall be the one indicated in preembassyHashAlgorithm
  • mimetypen:
    • MIME-Type of data associated with the reference ‘n’ in a manifest signature. If this property is not indicated, the ‘application/octet stream’ type shall be used.
  • contentTypeOidn:
    • OID or URN of the type of data signed for the reference number ‘n’ in a manifest signature. This property is complementary (non-exclusive) to the mimetypen property.
  • encodingn:
    • Coding of the data associated with the reference number ‘n’ in a manifest signature. Improper use of this property may lead to the generation of an invalid signature.
  • addKeyInfoKeyValue:
    • true: Includes the KeyValue node within KeyInfo from XAdES (default behavior).
    • false: Does not include the KeyValue node within XAdES KeyInfo.
  • addKeyInfoKeyName:
    • true: Includes the KeyName node within KeyInfo from XAdES.
    • false: Does not include the KeyName node within XAdES KeyInfo (default behavior).
  • avoidXpathExtraTransformsOnEnveloped:
    • true: Prevents the inclusion of the XPATH2 transformation that is normally added to enable co-signatures and removes all signatures from the document to leave only the content. WARNING: The co-signing of a document in which at least one of the signatures does not include the XPATH transformation will result in a signature document that will potentially be incorrectly validated by the signature validators. For this reason, the use of this property is only allowed in the signature operation (not in the co-signature operation).
    • false: Includes the XPATH2 transformation enables co-signatures by removing all signatures from the document to leave content alone (default behavior).
  • format:
    • XAdES Enveloping: Generates signatures in Enveloping format. This is the default format used when none is indicated.
    • XAdES Enveloped: Generates signatures in Enveloped format.
    • XAdES Detached: It generates signatures in Internally Detached format.
    • XAdES Externally Detached:Generates signatures in Externally Detached format.
  • includeOnlySignningCertificate:
    • true: Indicates that only the certificate of the signatory should be included in the signature.
    • false: Indicates that the entire certification chain of the signatory certificate should be included in the signature. Default value.
  • policyIdentifier:
    • Signature policy identifier (typically a URL to the policy in actionable XML format), needed to generate XAdES-EPES signatures.
  • policyIdentifierHash:
    • Base Chain 64 with the fingerprint of the signature policy. It is mandatory to indicate this property if it also indicated policyIdentifier, as it is also mandatory to give value to the property PolicyIdentifierHashAlgorithm.
  • policyIdentifierHashAlgorithm:
    • SHA1: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA1 algorithm.
    • SHA-256: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-256 algorithm.
    • SHA-384: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-384 algorithm.
    • SHA-512: It indicates that the fingerprint indicated in policyIdentifierHash property was calculated using the SHA-512 algorithm.
  • policyQualifier:
    • URL accessible to the document (usually PDF) containing a textual description of the signature policy. This property is optional even if you want to generate XAdES-EPES signatures.
  • policyDescription:
    • Textual description of the signature policy. If an XML is signed, the encoding of the used text should be aligned with the signed XML. This property is optional even if you want to generate XAdES-EPES signatures.
  • signerClaimedRoles:
    • He adds fields to the firm with the charges attributed to the signatory. The charges must be separated with the character "|" (and this cannot be in the text itself of any charge). If an XML is signed, the encoding of the used text should be aligned with the signed XML.
  • signatureProductionCity:
    • It adds to the firm a field with the city in which the signature is made. If an XML is signed, the encoding of the used text should be aligned with the signed XML.
  • signatureProductionProvince:
    • It adds to the firm a field with the province in which the signature is made. If an XML is signed, the encoding of the used text should be aligned with the signed XML.
  • signatureProductionPostalCode:
    • Add to the firm a field with the postal code in which the signature is made. If an XML is signed, the encoding of the used text should be aligned with the signed XML.
  • signatureProductionCountry:
    • Add to the firm a field with the country in which the signature is made. If an XML is signed, the encoding of the used text should be aligned with the signed XML.
  • referencesDigestMethod:
    • http://www.w3.org/2000/09/xmldsig#sha1: Use the SHA1 algorithm to calculate fingerprints of signed XML references.
    • http://www.w3.org/2001/04/xmlenc#sha256: Use the SHA-256 algorithm to calculate fingerprints of signed XML references.
    • http://www.w3.org/2001/04/xmlenc#sha512: Use the SHA-512 algorithm to calculate fingerprints of signed XML references. This is the default behavior.
  • mimeType:
    • MIME-Type of data to be signed. If this property is not indicated, the system tries to self-detect the type, establishing the closest (which may not be the strictly correct).
  • encoding:
    • URI with the encoding of the data to be signed (see the documentation of the XMLDSig Object element for more information). Improper use of this property may lead to the generation of an invalid signature. If data is provided to be previously coded in Base64 but is to be considered as its decoded form, this value should be set to http://www.w3.org/2000/09/xmldsig#base64 and the actual rate specified in the mimeType property. For example, to sign a PNG image by making the signature refer to its direct binary form, you can provide the image directly encoded in Base64 by indicating the encoding as http://www.w3.org/2000/09/xmldsig#base64 and the mimeType as image/png. The value must always be a URI.
  • outputXmlEncoding:
    • Output XML encoding. If this value is not indicated, an attempt is made to self-detect from the input XML (if the data to be signed is an XML).
  • contentTypeOid:
    • OID or URN type OID with the type identifier signed. This property is complementary (non-exclusive) to the mimeType property.
  • canonicalizationAlgorithm:
    • http://www.w3.org/TR/2001/REC-xml-c14n-20010315: The XML is signed with inclusive XML 1.0 canonized (default value).
    • http://www.w3.org/TR/2001/REC-xml-c14n-20010315#WithComments: The XML is signed with inclusive XML 1.0 canonized with comments.
    • http://www.w3.org/2001/10/xml-exc-c14n#: The XML is signed with exclusive XML 1.0 canonized.
    • http://www.w3.org/2001/10/xml-exc-c14n#WithComments: The XML is signed with exclusive XML 1.0 canonized with comments.
  • xadesNamespace:
    • http://uri.etsi.org/01903/v1.3.2#: URL for defining the XAdES namespace corresponding to version 1.3.2 of XAdES. This is the default value.
    • http://uri.etsi.org/01903/v1.4.1#: XAdES namespace definition URL corresponding to version 1.4.1 of XAdES. If this property is established you may also need to set the signedPropertyTypeUrl property to avoid inconsistencies in the XAdES version.
  • signedPropertiesTypeUrl:
    • http://uri.etsi.org/01903#SignedProperties: Signed Properties URL of XAdES. This is the default value.
    • http://uri.etsi.org/01903/1.3.2#SignedProperties: Signed Properties URL of XAdES v1.3.2.
    • http://uri.etsi.org/01903/1.4.1#SignedProperties: Signed Properties URL of XAdES v1.4.1. If this property is established it may be necessary to also establish the property xadesNamespace to avoid inconsistencies in the version of XAdES.
  • ignoreStyleSheets:
    • true: If you sign an XML with style sheets, ignore them by leaving them unsigned.
    • false: If an XML with style sheets is signed, also sign the style sheets (default value, see additional notes on signature style sheets).
  • avoidBase64Transforms:
    • true: It does not declare Base64 transformations even if necessary.
    • false: Declares Base64 transformations when the data to be signed in Base64 has been internally coded (default value).
  • headless:
    • true: Prevents further graphical dialogues to the user (e.g. for the deferencing of style sheets linked to relative routes).
    • false: Enables additional graphical dialogues to be displayed to the user.
  • xmlTransforms:
    • Number of transformations to be applied to the signed content. The same number of xmlTransformnType parameters should then be indicated, replacing n with a consecutive ordinal, starting at 0 (see additional notes on indication of additional transformations).
  • xmlTransformnType:
    • http://www.w3.org/2000/09/xmldsig#base64: It indicates that the data provided to sign is already encoded in Base64 and this additional transformation must be declared so that they are decoded before signing them. This Base64 transformation is in addition to the transformation needed to pass the data through the customer's signature methods.
    • http://www.w3.org/TR/1999/REC-xpath-19991116: The content must be processed through this XPATH transformation before it is signed. only applies when XML content is signed.
    • http://www.w3.org/2002/06/xmldsig-filter2: The content must be processed through this XPATH2 transformation before it is signed. only applies when XML content is signed.
  • xmlTransformnSubtype
    • Subtype of transformation n. The accepted values and their functionality depend on the value indicated in xmlTransformnType.
  • xmlTransformnBody:
    • Body of transformation n. The accepted values and their functionalities depend on the values indicated in xmlTransformnType and xmlTransformnSubtype.
  • nodeToSign:
    • Node identifier (established by the attribute "Id") to be signed within an XML.
  • commitmentTypeIndications:
    • Number of commitmentTypeIndications to be declared. These are the reasons for the signing. Specific values are specified with commitmentTypeIndicationnIdentifier and commitmentTypeIndicationnDescription, where ‘n’ goes from 0 to the minus 1 value indicated on this property.
  • commitmentTypeIndicationnIdentifier:
    • 1: It states that the number n (counting from scratch) CommitmentTypeIndications is "Proof of Origin".
    • 2: It states that the number n (counting from scratch) CommitmentTypeIndications is "Reception Test".
    • 3: It states that the number n (counting from scratch) CommitmentTypeIndications is "Delivery Test".
    • 4: Sets that the number n (counting from scratch) is "Shipping Test".
    • 5: It states that the number n (counting from scratch) CommitmentTypeIndications is "Approval Test".
    • 6: Establishes that the number n (counting from scratch) CommitmentTypeIndications is "Proof of Creation".
  • commitmentTypeIndicationnDescription:
    • Sets the description of the StringTypeIndications number n. This attribute is optional.
  • commitmentTypeIndicationnDocumentationReferences:
    • List of URLs separated by '|' character that are provided as documentary references of the CommitmentTypeIndication number n (optional attribute). List URLs cannot contain '|' character (as this is used as a separator).
  • TypeDesignDesignDesignQualifiers:
    • List of textual indicators separated by '|' character that are provided as additional qualifiers of the DiffTypeIndication number n (optional attribute). They're usually OID. The items in the list cannot contain the '|' character (as it is used as a separator).
  • allowSignLTSignature:
    • true: Longevo file signatures are allowed to be co-signed/countersigned, even though they will not be valid thereafter.
    • false (or other value): Co-signing/counter-signing of longevo file signatures is not permitted.
  • avoidAGEPolicyIncompatibilities:

    • false (or other value): Co-signing with AGE's signature policy is not allowed when the result will not be compatible with the policy.
  • allowXadesEnvelopedWithoutXPath:
    • true: Allows the co-signature of a signature that does not contain an XPath transform and will therefore be invalidated. The XPath transformation will be included in the new co-signature.
    • false: Co-signing of signatures that do not contain an XPath transform is not allowed.

PAdES sign-up options

The configuration properties supported by PAdES signatures are listed below.

  • includeOnlySignningCertificate:
    • true: Indicates that only the certificate of the signatory should be included in the signature.
    • false: Indicates that the entire certification chain of the signatory certificate should be included in the signature. Default value.
  • alwaysCreateRevision:
    • true: You will always create a review when you sign. Requires the document to comply with PDF 1.7 specification (ISO 32000-1:2008)
    • false: It will not create revision at the first firm and yes at the following.
  • signatureField:
    • Name of the pre-existing signature field in which the signature is inserted.
  • signaturePages:
    • Indicates the pages on which the signature must appear visible.
    • Supports the following values:
      • all: In order to indicate that the visible signature appears on all pages of the document, it must be given this value.
      • append: If the visible signature must appear on a new blank page added at the end of the document, it must be given this value.
      • Page or page listing: This option allows you to select the page or page range where the visible signature will appear. If you want to indicate a page or several pages where the visible signature is displayed, you can indicate page by page with a comma separation (1,4.7: It is stamped on pages 1, 4 and 7 or indicating a range of pages through a script (1-8: It is printed on the pages from 1 to 8). As explained in the previous paragraph, it is permitted to indicate the pages with a negative value, where it will start from the end (1,-1: It will be printed on the first and last page).
  • signaturePositionOnPageLowerLeftX:
    • Whole number with horizontal coordinate from the lower left corner of the page to the lower left corner of the visible signature field.
  • signaturePositionOnPageLowerLeftY:
    • Whole number with vertical coordinate from the lower left corner of the page to the lower left corner of the visible signature field.
  • signaturePositionOnPageUpperRightX:
    • Whole number with horizontal coordinate from the lower left corner of the page to the upper right corner of the visible signature field.
  • signaturePositionOnPageUpperRightY:
    • Whole number with vertical coordinate from the lower left corner of the page to the upper right corner of the visible signature field.
  • signatureRubricImage:
    • Path of the file with the JPEG image of the image to be displayed in the visible signature PDF. Base 64 of the image can also be directly indicated.
  • layer2Text:
    • Text to show in the visible signature field.
  • layer2FontFamily:
    • 0: The text of the visible signature will be displayed with Courier source. This is the default value.
    • 1: The text of the visible signature will be displayed with Helvetica source.
    • 2: The text of the visible signature will be displayed with Times Roman font.
    • 3: The text of the visible signature will be displayed with Symbol font.
    • 4: The text of the visible signature will be displayed with ZapfDingBats source.
  • layer2FontSize:
    • Font size of visible signature text.
  • layer2FontStyle:
    • 0: Text of the visible signature without style. Default value.
    • 1: Text of the signature visible in bold.
    • 2: Text of the visible signature in italics.
    • 4: Text of the underlined visible signature.
    • 8: Text of the marked visible signature.
  • layer2FontColor:
    • black: Text of the visible signature without style. Default value.
    • white: The text of the visible signature shall be white.
    • gray: The text of the visible signature shall be grey.
    • lightGray: The text of the visible signature shall be light grey.
    • darkgray: The text of the visible signature shall be dark grey.
    • network: The text of the visible signature shall be red.
    • pink: The text of the visible signature shall be pink.
  • obfuscateCertText:
    • true: User identifiers extracted from the CN or DN of the certificate and displayed in the visible signature PDF are offended. Data from pseudonym certificates are not offended. This is the default value.
    • false: The information on the certificates is not offended.
  • obfuscationMask:
    • Criteria for the obfuscation of user identifiers in visible signatures PDF. You must show the following pattern:
      • character; longitudinal digits; positions; displacement
    • In this pattern:
      • caracter: It's the character to use to offend characters.
      • longitudinally digits: Minimum number of digits a text string must have in order to be deemed to be offended.
      • positions: List of positions indicating which characters should be displayed. The listing is expressed with a sequence of true/false separated by commas (','), where true indicates that the character must be displayed and false than not.
      • displacement: Indicates whether the displacement of positions of the mask is allowed to show all the characters indicated (true) or whether it must be respected (false).
  • visibleSignature:
    • default: Visible signature PDF will be made if the parameters have been provided with the area and the signature page. This is the default value.
    • want: The user must select the visible signature area. If you cancel the process:
      • If the petition also includes visible signature area parameters (position and page), these parameters will be used and the signing process will continue.
